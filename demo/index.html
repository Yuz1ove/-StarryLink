StarryLink

星夜 StarryLink

海纜・地網・衛星多路徑災害韌性通訊助理

星夜是一套面向災害弱網環境的韌性通訊系統。當地震、颱風或海纜／地面骨幹網路異常造成封包遺失、延遲升高或通訊中斷時，系統會將使用者狀態壓縮成低資料量封包，優先傳送 GPS、求救等級、生命狀態與按鍵回覆，並依照通訊品質自動選擇 Wi-Fi、5G、簡訊、BLE 中繼或衛星備援路徑，協助守望隊快速判斷誰需要優先救援。

Competition Brief

海、地、星空三層路徑下，先讓最小必要訊息送達

這裡的「海」指的是海纜、國際骨幹與跨區網路連線，不是海上救援。星夜把受困者端操作壓到最少，並把後台需要判斷的風險與通訊路徑整理成可追蹤、可重送、可解釋的事件資料。

海

海纜與骨幹網路異常偵測

當跨區連線、骨幹延遲或封包遺失率異常時，系統會判定外部網路不穩，降低資料量並啟用備援策略。

  • 監測海纜、國際骨幹、區域閘道與外部 API 可用性。
  • 海纜/骨幹異常時暫停非必要 UI 同步。
  • 優先傳送最低可行求救封包。
地

地面弱網通訊與守望隊調度

以 Wi-Fi、5G、基地台、BLE 中繼與本地節點作為主要通訊層，負責手機端狀態回報、GPS 定位與守望隊後台監測。

  • 手機端只需大按鈕回覆安全、受傷、被困或需要救援。
  • 後台依風險分數排序並建議主動聯繫或派遣。
星

衛星與高風險備援通道

當地面網路失效或高風險個案無法回傳狀態時，系統會將衛星或其他高成本通道保留給真正需要優先救援的個案。

  • Red 風險會優先傳送 GPS、求救狀態與生命狀態。
  • 衛星通道在一般狀況保持低權重，高風險或地面失效時升權。

Sea・Ground・Space

海地星空通訊架構

星夜不是單一路徑通報系統，而是根據災害環境自動切換通訊路徑的韌性調度系統。

A

Sea Layer

海 Sea Layer

負責判斷跨區連線是否異常，例如延遲升高、封包遺失率上升、外部 API 無法連線。當此層異常時，星夜會降低資料負載，減少非必要同步，改送最低可行求救封包。

  • Submarine Cable / 海纜
  • International Backbone / 國際骨幹
  • Regional Gateway / 區域閘道
  • Backbone Health Monitor / 骨幹健康監測
B

Ground Layer

地 Ground Layer

這是星夜主要運作層。手機端會透過地面網路傳送低資料量狀態包，守望隊後台根據封包成功率、GPS、使用者回覆與風險分數進行調度。

  • 5G / LTE
  • Wi-Fi
  • Local Mesh / BLE Relay
  • SMS Fallback
  • Rescue Console / 守望隊工作台
  • GPS Status Packet
C

Space Layer

星空 Space Layer

當地面網路不穩且個案風險分數過高時，星夜會將高成本備援通道保留給高風險使用者，例如無回覆、GPS 靜止、健康狀態異常、封包多次失敗者。

  • Satellite Backup
  • Emergency Uplink
  • High-Risk Priority Channel

地震後海纜與地面骨幹不穩情境

地震後，區域地面基地台壅塞，部分海纜或骨幹網路延遲升高，使用者手機無法穩定傳送完整訊息。星夜會自動改用低資料量封包，只傳送 GPS、求救狀態、生命狀態與按鍵回覆，並在守望隊後台即時顯示受困者風險分數與建議通訊路徑。

決策引擎

背後程式運行邏輯

從手機端按鈕到守望隊排序，星夜把每一步都拆成可觀察的低資料量封包、通訊矩陣、ACK、retry、去重與風險加權。

1

手機端產生狀態

受困者在手機端按下最少量的狀態按鈕。

  • 我安全
  • 我受傷
  • 我被困住
  • 我需要醫療
  • 無法說話
  • 需要救援
2

低資料量封包產生

手機端把狀態壓縮為最小必要欄位。

{
  userId: "U-DEMO",
  timestamp: 1780000000000,
  gps: { lat: 25.035, lng: 121.564, accuracy: "high" },
  statusCode: "NEED_HELP",
  batteryLevel: 72,
  riskLevel: "ORANGE",
  packetSeq: 1842,
  retryCount: 1,
  checksum: "7a31"
}
3

根據通訊矩陣選擇通道

系統依封包成功率、延遲、訊號、GPS、成本與電量影響計算通道分數。

  1. Green / Yellow：Wi-Fi、5G 或 SMS 優先
  2. Orange：低資料量封包與重送頻率提高
  3. Red：必要時啟用衛星或高優先備援通道
4

守望隊後台接收

封包傳送至守望隊工作台後，同步更新受困者清單與事件細節。

  • 最新回覆
  • 最後同步時間
  • GPS 狀態
  • 封包成功率與建議通訊路徑
  • 是否需要人工確認
5

風險評估引擎

依據矩陣加權，形成可解釋的風險分數。

  • 使用者是否回覆、是否按下求救
  • GPS 靜止、封包連續失敗、電量過低
  • 受傷、呼吸困難、被困、失溫與最後同步時間
6

升級通訊路徑或救援優先級

最後輸出風險等級並決定後續處理方式。

RED ORANGE YELLOW GREEN
  • Red：優先派遣或升級衛星備援
  • Orange：守望隊主動確認
  • Yellow：持續觀察與低成本通道
  • Green：維持平時監測

Low Data Packet

低資料量封包與降低丟包率邏輯

星夜不是把完整文字硬送出去，而是把狀態壓縮成短碼與必要欄位，並用 seq、retry 與 checksum 提高弱網環境下的可追蹤性。

策略 做法 降低丟包/錯誤的原因 說明
狀態碼 完整文字改成 OK / NEED_HELP / INJURED / TRAPPED / NO_RESPONSE payload 變小，弱網下較容易送達 手機端只需要按大按鈕
最小必要資訊 優先送 GPS、求救狀態、生命狀態、電量與時間 網路差時暫停非必要 UI 同步 後台仍可排序與派遣
packetSeq 每包都有連續序號 可追蹤順序並去重 重送同一 seq 不會被誤判成多次求救
retryCount ACK pending 時增加重送次數 封包連續失敗會回饋到風險分數 高風險時提高重送優先級
checksum 簡化驗證避免錯誤資料 後台能辨識資料是否被破壞 Demo 使用輕量 checksum，不宣稱正式加密
通訊矩陣 依成功率最高且符合風險的通道送出 地網失效時改用 SMS、BLE 或衛星備援 衛星只留給高風險或地面失效者

A

低資料量封包範例

type RescuePacket = {
  userId: string
  timestamp: number
  gps: { lat: number | null; lng: number | null; accuracy: string }
  statusCode: "OK" | "NEED_HELP" | "INJURED" | "TRAPPED" | "NO_RESPONSE"
  batteryLevel: number
  riskLevel: "GREEN" | "YELLOW" | "ORANGE" | "RED"
  packetSeq: number
  retryCount: number
  checksum: string
}

function encodeLowDataPacket(userState) {
  return {
    userId: userState.id,
    timestamp: Date.now(),
    gps: userState.gps,
    statusCode: mapStatus(userState.reply),
    batteryLevel: userState.battery,
    riskLevel: userState.riskLevel,
    packetSeq: nextSeq(),
    retryCount: userState.retryCount,
    checksum: checksum(...)
  }
}
B

通訊路徑評分

function calculateChannelScore(channel, userRisk) {
  let score =
    channel.packetSuccessRate * 0.35 +
    channel.latencyScore * 0.20 +
    channel.signalStrength * 0.15 +
    channel.gpsAvailability * 0.15 +
    channel.channelCost * 0.10 +
    channel.batteryImpact * 0.05

  if (channel.id === "SATELLITE" && userRisk === "RED") score += 18
  if (channel.id === "SATELLITE" && userRisk === "GREEN") score -= 16

  return clamp(score, 0, 100)
}
C

風險分數矩陣

function calculateRiskScore(user) {
  let score = 0

  if (!user.replied) score += 26
  if (user.statusCode === "NEED_HELP") score += 38
  if (user.statusCode === "INJURED") score += 42
  if (user.statusCode === "TRAPPED") score += 54

  if (user.gpsStaticMinutes >= 10) score += 20
  if (user.retryCount >= 3 || user.ackStatus === "failed") score += 24
  if (user.batteryLevel < 20) score += 16
  if (user.breathingDifficulty) score += 28
  if (user.minutesSinceLastSync >= 8) score += 10

  return Math.min(score, 100)
}

function riskLevel(score) {
  if (score >= 80) return "RED"
  if (score >= 55) return "ORANGE"
  if (score >= 25) return "YELLOW"
  return "GREEN"
}
