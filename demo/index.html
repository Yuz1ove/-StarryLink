星夜｜AI 韌性通訊與受困者狀態確認系統

在災害或弱訊號環境下，星夜用低資料量封包、ACK 確認、序列去重、retry 重送與 fallback route，協助守望隊確認受困者狀態，降低訊息遺失風險，並用風險矩陣決定優先處理順序。

Competition Brief

災害弱訊號下，先確認人是否安全

星夜把受困者端的操作壓到最少，把後台需要判斷的狀態整理成可追蹤、可重送、可解釋的事件資料，讓展示焦點回到「訊息能否送達、誰需要先被確認」。

A

星夜是什麼

星夜不是一般聊天 App，也不是複雜醫療系統。它是一套「災害弱訊號下的狀態確認 MVP」。

  • 受困者只需要按下最少量的狀態按鈕，例如：我安全、我不舒服、無法回覆。
  • 系統把這些狀態轉成極小資料封包送到守望隊後台。
  • 守望隊根據回覆狀態、訊號品質、GPS、最後同步時間與風險分數決定誰要優先處理。
B

三分鐘 Demo 做什麼

Demo 會模擬 5 位目標人物，其中 4 位依照預設事件流程自動跑，1 位作為手機實際操作端 Demo。

  • 守望隊後台會即時收到狀態、ACK、風險變化與優先順序。
  • 展示重點不是華麗動畫，而是弱訊號下如何仍然確認人是否安全。
C

決策引擎與低資料通訊

前端傳送的不是長文字，而是低資料量狀態碼；每個封包具有 seq、clientId、statusCode、gps、signalQuality、timestamp。

  • 後台透過 ACK 確認有無收到；ACK pending 時會 retry。
  • 重複封包用 seq 去重，弱訊號則改用 fallback route 或低頻同步。
  • 風險分數不是 AI 黑盒，而是可解釋的矩陣加權。

決策引擎

系統整體邏輯

從手機端按鈕到守望隊排序，星夜把每一步都拆成可觀察的封包、ACK、retry、去重與風險加權。

1

使用者狀態輸入

受困者在手機端按下最少量的狀態按鈕。

  • 我安全
  • 我身體不適
  • 我需要協助
  • 無法回覆 / 逾時
2

低資料量封包產生

手機端把狀態轉成短碼與必要欄位。

{
  clientId: "U-013",
  seq: 1842,
  statusCode: "UNWELL",
  gps: { lat: 22.997, lng: 120.212 },
  signalQuality: 32,
  timestamp: 1720000000000
}
3

傳輸可靠化處理

封包盡量小，但仍保留送達確認與弱訊號備援。

  1. 壓縮 payload
  2. 加入 seq
  3. 發送封包
  4. 等待 ACK
  5. ACK pending 則 retry
  6. 收到重複 seq 則 dedupe
  7. 弱訊號則 fallback route
4

守望隊後台接收

後台同步更新受困者清單與事件細節。

  • 最新回覆
  • 最後同步時間
  • GPS 狀態
  • 通訊品質
  • 是否 ACK pending
  • 是否需要人工確認
5

風險評估引擎

依據矩陣加權，形成可解釋的風險分數。

  • 使用者回覆、身體狀態、訊號品質
  • GPS 可用性、最後同步時間
  • 是否無法回覆、是否多次 retry 失敗
6

優先順序排序

最後輸出等級並決定後續處理方式。

CRITICAL HIGH MEDIUM LOW
  • 立即派守望隊確認
  • 發送簡訊或語音 IVR
  • 繼續監測
  • 標記為安全

程式運行

可讀的程式邏輯展示區

這裡不打開原始碼檔案，而是把 Demo 背後的判讀方式整理成評審可快速閱讀的 TypeScript-like 邏輯。

A

低資料量封包範例

type RescuePacket = {
  clientId: string
  seq: number
  statusCode: "SAFE" | "UNWELL" | "HELP" | "NO_RESPONSE"
  gps?: { lat: number; lng: number }
  signalQuality: number
  timestamp: number
}

function buildPacket(input) {
  return {
    clientId: input.clientId,
    seq: nextSeq(),
    statusCode: input.statusCode,
    gps: input.gps,
    signalQuality: input.signalQuality,
    timestamp: Date.now()
  }
}
B

ACK / retry / seq 去重邏輯

function handleIncomingPacket(packet) {
  if (seenSeq.has(packet.clientId + ":" + packet.seq)) {
    return { ignored: true, reason: "duplicate-seq" }
  }

  seenSeq.add(packet.clientId + ":" + packet.seq)
  sendAck(packet.clientId, packet.seq)
  updateClientState(packet)
}

function shouldRetry(packetState) {
  return packetState.ackStatus === "pending" && packetState.retryCount < 3
}
C

風險分數矩陣

function calculateRiskScore(user) {
  let score = 0

  if (user.statusCode === "HELP") score += 30
  if (user.statusCode === "UNWELL") score += 35
  if (user.statusCode === "CANNOT_MOVE") score += 45
  if (user.statusCode === "NO_RESPONSE") score += 50

  if (user.signalQuality < 40) score += 25
  if (!user.gpsConfirmed) score += 25
  if (user.ackStatus === "pending") score += 15
  if (user.minutesSinceLastSync > 10) score += 35

  return Math.min(score, 100)
}

function riskLevel(score) {
  if (score >= 80) return "CRITICAL"
  if (score >= 50) return "HIGH"
  if (score >= 25) return "MEDIUM"
  return "LOW"
}
